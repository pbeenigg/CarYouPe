# Contexts 使用手册

## 概述

Contexts（上下文模式）定义了 AI 助手的工作模式和行为偏好。不同的上下文适用于不同类型的工作任务。

## 上下文作用

选择合适的上下文可以：

- 调整 AI 助手的行为优先级
- 优化工具使用偏好
- 改变输出风格和格式
- 提高特定任务的效率

## 可用上下文

### 1. dev (开发模式)

**模式**：主动开发
**焦点**：实现、编码、构建功能

#### 行为特点

- 先写代码，后解释
- 优先工作方案而非完美方案
- 更改后运行测试
- 保持提交原子性

#### 优先级

1. 让它能工作 (Get it working)
2. 让它正确 (Get it right)
3. 让它干净 (Get it clean)

#### 偏好工具

- `Edit`、`Write` - 代码更改
- `Bash` - 运行测试/构建
- `Grep`、`Glob` - 查找代码

#### 适用场景

- 开发新功能
- 修复 bug
- 实现业务逻辑
- 日常编码工作

---

### 2. research (研究模式)

**模式**：探索、调查、学习
**焦点**：在行动前充分理解

#### 行为特点

- 先广泛阅读再下结论
- 提出澄清性问题
- 边探索边记录发现
- 理解清楚前不写代码

#### 研究流程

1. 理解问题
2. 探索相关代码/文档
3. 形成假设
4. 用证据验证
5. 总结发现

#### 偏好工具

- `Read` - 理解代码
- `Grep`、`Glob` - 查找模式
- `WebSearch`、`WebFetch` - 外部文档
- `Task` with Explore agent - 代码库问题

#### 输出格式

发现优先，建议其次

#### 适用场景

- 调查问题
- 理解现有代码
- 学习新技术
- 分析可行性

---

### 3. review (审查模式)

**模式**：PR 审查、代码分析
**焦点**：质量、安全、可维护性

#### 行为特点

- 评论前彻底阅读
- 按严重性排序（关键 > 高 > 中 > 低）
- 建议修复方案，而非仅指出问题
- 检查安全漏洞

#### 审查清单

- [ ] 逻辑错误
- [ ] 边缘情况
- [ ] 错误处理
- [ ] 安全（注入、认证、密钥）
- [ ] 性能
- [ ] 可读性
- [ ] 测试覆盖

#### 输出格式

按文件分组，严重性优先

#### 适用场景

- 代码审查
- PR 评审
- 安全审计
- 质量检查

---

## 如何使用上下文

### 通过会话设置

在开始工作前，可以声明当前上下文：

```
我现在进入开发模式，需要实现用户认证功能
```

或

```
进入研究模式，帮我调查一下这个性能问题
```

### 上下文切换

工作过程中可以切换上下文：

```
# 开始时研究问题
我先需要研究一下现有的认证实现...

# 理解后切换到开发
好的，现在开始实现新功能...

# 完成后切换到审查
帮我审查一下刚才写的代码
```

## 上下文组合

一个完整的任务通常包含多个上下文阶段：

```
研究 (research) → 开发 (dev) → 审查 (review)
     ↓                ↓             ↓
  理解问题        实现方案      检查质量
```

## 自定义上下文

创建新上下文的步骤：

1. 在 `.agent/contexts/` 目录创建 `.md` 文件
2. 定义模式、焦点和行为
3. 指定工具偏好

**模板**：

```markdown
# 上下文名称

Mode: 模式描述
Focus: 焦点描述

## Behavior

- 行为 1
- 行为 2
- 行为 3

## Priorities

1. 优先级 1
2. 优先级 2
3. 优先级 3

## Tools to favor

- 工具 1 for 用途
- 工具 2 for 用途

## Output

输出格式说明
```

## 最佳实践

1. **明确声明上下文** - 开始工作前说明当前模式
2. **适时切换** - 根据任务阶段切换合适的上下文
3. **保持一致性** - 在同一上下文内保持行为一致
4. **善用组合** - 复杂任务使用多个上下文阶段
