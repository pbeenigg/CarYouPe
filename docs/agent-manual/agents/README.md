# Agents 使用手册

## 概述

Agents 是预定义的专业 AI 角色，每个 Agent 具有特定的专长领域和工作方式。它们可以被命令调用，也可以在合适的场景下主动激活。

## Agent 配置格式

每个 Agent 配置文件使用 YAML frontmatter 定义元数据：

```markdown
---
name: agent-name
description: Agent 描述
tools: Read, Grep, Glob, Bash # 可用工具
model: opus # 使用的模型
---

# Agent 内容

...
```

## 可用 Agents 列表

### 1. architect (架构师)

**用途**：系统架构设计、可扩展性分析、技术决策

**何时使用**：

- 规划新功能时
- 重构大型系统时
- 做架构决策时

**主要职责**：

- 设计系统架构
- 评估技术权衡
- 推荐模式和最佳实践
- 识别可扩展性瓶颈
- 规划未来增长

**示例**：

```
请帮我设计一个实时通知系统的架构
```

---

### 2. planner (规划师)

**用途**：创建全面的、可操作的实施计划

**何时使用**：

- 开始新功能开发时
- 进行重大架构变更时
- 复杂重构工作时

**主要职责**：

- 分析需求
- 分解功能为可管理的步骤
- 识别依赖关系和潜在风险
- 建议最佳实施顺序

**计划格式**：

```markdown
# 实施计划: [功能名称]

## 概述

[2-3 句摘要]

## 需求

- [需求 1]
- [需求 2]

## 实施步骤

### 阶段 1: [阶段名]

1. **[步骤名]** (文件: path/to/file.ts)
   - 操作: 具体操作
   - 原因: 步骤原因
   - 依赖: 无 / 需要步骤 X
   - 风险: 低/中/高

## 测试策略

- 单元测试: [测试文件]
- 集成测试: [测试流程]
- E2E 测试: [用户流程]

## 风险与缓解

- **风险**: [描述]
  - 缓解: [如何处理]

## 成功标准

- [ ] 标准 1
- [ ] 标准 2
```

### 3. code-reviewer (代码审查员)

**用途**：代码质量检查、安全审查、可维护性评估

**何时使用**：

- 编写或修改代码后立即使用
- 所有代码变更都应该经过审查

**检查清单**：

**安全检查 (关键)**：

- [ ] 硬编码凭据
- [ ] SQL 注入风险
- [ ] XSS 漏洞
- [ ] 缺少输入验证
- [ ] 不安全的依赖

**代码质量 (高)**：

- [ ] 大函数 (>50 行)
- [ ] 大文件 (>800 行)
- [ ] 深层嵌套 (>4 级)
- [ ] 缺少错误处理
- [ ] console.log 语句

**性能 (中)**：

- [ ] 低效算法
- [ ] 不必要的重渲染
- [ ] 缺少缓存
- [ ] N+1 查询

**输出格式**：

```
[严重] 硬编码 API 密钥
文件: src/api/client.ts:42
问题: API 密钥暴露在源代码中
修复: 移动到环境变量

const apiKey = "sk-abc123";  // ❌ 错误
const apiKey = process.env.API_KEY;  // ✓ 正确
```

### 4. build-error-resolver (构建错误解决器)

**用途**：诊断和修复构建错误

**何时使用**：

- `npm run build` 失败时
- TypeScript 编译错误时
- 依赖问题时

**处理流程**：

1. 分析构建错误输出
2. 识别根本原因
3. 提供修复方案
4. 验证修复有效

---

### 5. doc-updater (文档更新器)

**用途**：更新项目文档、API 文档、README

**何时使用**：

- 添加新功能后
- API 变更后
- 需要更新文档时

---

### 6. e2e-runner (E2E 测试运行器)

**用途**：端到端测试编写和运行

**何时使用**：

- 验证用户流程时
- 完成功能开发后
- 回归测试时

---

### 7. refactor-cleaner (重构清理器)

**用途**：代码重构和清理

**何时使用**：

- 需要重构代码时
- 清理技术债务时
- 改善代码结构时

---

### 8. security-reviewer (安全审查员)

**用途**：安全漏洞检查和最佳实践审查

**何时使用**：

- 实现认证/授权时
- 处理用户输入时
- 处理敏感数据时
- 创建 API 端点时

---

### 9. tdd-guide (TDD 指导)

**用途**：测试驱动开发指导

**何时使用**：

- 实现新功能时
- 修复 bug 时
- 重构代码时

**TDD 循环**：

```
RED → GREEN → REFACTOR → REPEAT

RED:      编写失败的测试
GREEN:    编写最少代码使测试通过
REFACTOR: 改进代码，保持测试通过
REPEAT:   下一个功能/场景
```

## 如何调用 Agent

Agents 通常通过命令调用，例如：

```
/plan 实现用户认证功能    # 调用 planner
/code-review              # 调用 code-reviewer
/tdd 创建订单处理函数      # 调用 tdd-guide
```

## 自定义 Agent

创建新 Agent 的步骤：

1. 在 `.agent/agents/` 目录创建 `.md` 文件
2. 添加 YAML frontmatter 定义元数据
3. 编写 Agent 的职责和行为说明

**示例模板**：

```markdown
---
name: my-custom-agent
description: 自定义 Agent 描述
tools: Read, Grep, Glob
model: sonnet
---

# 自定义 Agent

## 职责

- 职责 1
- 职责 2

## 工作流程

1. 步骤 1
2. 步骤 2
```
